# n8n self-hosted — setup e decisões

**Data:** 2026-05-25
**Status:** ativo
**Stack:** n8n 2.12 + Cloudflare Tunnel + Docker Compose, na VPS Contabo

---

## Visão geral

n8n self-hosted exposto em `https://n8n.ifops.com.br` via Cloudflare Tunnel,
rodando na mesma rede Docker (`senses_default`) do microserviço
`senses-os-api`. Persistência via SQLite em volume Docker. Sem reverse-proxy
local — o tunnel termina TLS no edge da Cloudflare e fala HTTP plain com o
container `n8n` pela rede interna.

```
┌──────────────┐    HTTPS    ┌──────────────────┐   QUIC   ┌────────────────────────┐
│  Cliente Web │ ─────────▶  │ Cloudflare edge  │ ◀───────▶│ cloudflared-n8n        │
└──────────────┘             │ (n8n.ifops.com)  │  tunnel  │ (rede senses_default)  │
                             └──────────────────┘          └────────────┬───────────┘
                                                                        │ HTTP
                                                                        ▼
                                                          ┌──────────────────────────┐
                                                          │ n8n:5678                 │
                                                          │ senses-os-api:3000       │
                                                          └──────────────────────────┘
```

## Decisões arquiteturais

### Por que SQLite local (e não Postgres)

- Volume de execuções esperado é baixo (algumas dezenas de OS por dia).
- Operação single-node: não precisa de Postgres compartilhado.
- Menos um serviço pra manter, monitorar e fazer backup separado.
- Backup do volume Docker `n8n_data` cobre 100% do estado.
- Se em algum momento o load justificar, migrar pra Postgres é uma mudança
  isolada de env-vars + dump/restore.

### Por que Cloudflare Tunnel (e não Nginx + Let's Encrypt)

- Não precisa abrir portas 80/443 no firewall da VPS.
- Certificado, DDoS, WAF e CDN ficam no edge da Cloudflare sem config local.
- IP da VPS não fica exposto publicamente.
- Re-IP da VPS não quebra o serviço (tunnel reconecta sozinho).

### Por que `config.yml` local (e não config pelo painel da Cloudflare)

O painel Zero Trust > Tunnels > Public Hostnames *parece* salvar as ingress
rules, mas o `cloudflared` moderno, quando rodado com `--token`, **não
recebe** essas regras automaticamente em todas as situações — comportamento
inconsistente quando o tunnel é criado pelo dashboard mas o conector roda
em outra org/account ou em modo "remotely managed" desligado. Sintoma:
4 conexões registradas com sucesso + log `No ingress rules were defined ...
will return 503 for all incoming HTTP requests` + HTTP 503 público.

**Solução:** versionar `cloudflared/config.yml` no repo e montar como volume
read-only. O `cloudflared` lê localmente, ignora o painel. Vantagens:

- Auditoria: histórico das rotas vive no git.
- Reprodutibilidade: subir o mesmo tunnel em outra máquina é só
  `docker compose up` com o token.
- Resiliência a mudanças de UI da Cloudflare.

**Limitação:** alterar rotas exige `docker compose restart cloudflared`.
Não dá pra editar via painel.

### Por que basic auth nativo do n8n não está mais ativo

Os env-vars `N8N_BASIC_AUTH_*` foram **removidos do n8n 2.x**. A imagem
2.12 atual ignora silenciosamente (não loga deprecation, mas não aplica).
A proteção de acesso é via **owner setup** (User Management): a primeira
visita em `https://n8n.ifops.com.br/` exige criar a conta owner; depois
disso, qualquer requisição não-autenticada é redirecionada pra `/signin`.

**Implicação prática:**
- `curl https://n8n.ifops.com.br/` retorna HTTP 200 com o SPA do n8n (que
  roteia client-side pra `/setup` ou `/signin`). Não retorna 401.
- `curl https://n8n.ifops.com.br/healthz` retorna 200 público (endpoint
  intencionalmente aberto pra healthcheck).
- APIs e webhooks exigem auth de acordo com a config do workflow.

**Próxima ação na primeira visita ao painel:** completar o owner setup
com `igor@ifops.com.br` + senha forte (a mesma que estava prevista pra
basic auth pode ser usada).

## Arquivos sensíveis (nenhum versionado)

| Arquivo | Local | Permissão | Conteúdo |
|---|---|---|---|
| Tunnel token | `/root/.cloudflared/senses-contabo.token` | `600` | JWT do Cloudflare Tunnel (autentica o conector) |
| Encryption key | `/root/.n8n-encryption-key` | `600` | 64 hex chars; cifra credenciais salvas no n8n |
| Env do compose | `/root/senses/.env.n8n` | `600` | Token + encryption key + senha owner; lido por `docker compose --env-file` |

`.env.n8n` está coberto pelo `.gitignore` via padrão `.env.*` (com exceção
de `.env.example`).

## Comandos operacionais

```bash
# Subir o stack inteiro (n8n + cloudflared)
docker compose -f docker-compose.n8n.yml --env-file .env.n8n up -d

# Subir só o cloudflared (após editar config.yml)
docker compose -f docker-compose.n8n.yml --env-file .env.n8n up -d \
  --force-recreate cloudflared

# Restart sem recriar
docker compose -f docker-compose.n8n.yml restart cloudflared
docker compose -f docker-compose.n8n.yml restart n8n

# Logs
docker logs n8n --tail 50 -f
docker logs cloudflared-n8n --tail 50 -f

# Ver workflows persistidos
docker exec n8n ls /home/node/.n8n
```

## Adicionar nova rota pública pelo tunnel

1. Editar `cloudflared/config.yml`, adicionar nova entrada **antes** do
   `service: http_status:404` (catch-all):

   ```yaml
   ingress:
     - hostname: n8n.ifops.com.br
       service: http://n8n:5678
       originRequest:
         noTLSVerify: true
         connectTimeout: 30s
     - hostname: outro.ifops.com.br      # ← novo
       service: http://outro-svc:8080
     - service: http_status:404
   ```

2. Garantir que o CNAME `outro.ifops.com.br` aponta pro tunnel
   (`<tunnel-id>.cfargotunnel.com`, proxied) no DNS da Cloudflare.

3. `docker compose -f docker-compose.n8n.yml restart cloudflared`

4. Verificar logs: não pode aparecer `No ingress rules were defined`.

## Backup e restore

### Backup do volume

```bash
docker run --rm \
  -v senses_n8n_data:/data \
  -v /root/backup:/backup \
  alpine tar czf /backup/n8n-$(date +%Y%m%d).tar.gz -C /data .
```

Cobre: workflows, credenciais (cifradas com `N8N_ENCRYPTION_KEY`),
execuções, configs do owner. Sem a encryption key, credenciais ficam
ilegíveis — guardar `N8N_ENCRYPTION_KEY` separado do backup é
proposital pra reduzir blast radius.

### Restore

```bash
docker compose -f docker-compose.n8n.yml down
docker run --rm \
  -v senses_n8n_data:/data \
  -v /root/backup:/backup \
  alpine sh -c "cd /data && tar xzf /backup/n8n-YYYYMMDD.tar.gz"
docker compose -f docker-compose.n8n.yml --env-file .env.n8n up -d
```

Restaurar com a **mesma** `N8N_ENCRYPTION_KEY` que cifrou. Sem ela, todas
as credenciais salvas viram lixo.

## Rotação de credenciais

### Owner password (n8n)

1. Login no painel → Settings → Users → editar usuário → trocar senha.
2. Atualizar password manager pessoal.
3. `.env.n8n` **não** guarda mais a senha owner (n8n 2.x).

### Encryption key

⚠️ **Não rotacionar sem migrar** — invalida todas as credenciais salvas.
Procedimento:

1. Exportar todas as credenciais (Settings → Credentials → export).
2. `docker compose down`.
3. Gerar nova key, atualizar `.env.n8n` + `/root/.n8n-encryption-key`.
4. Apagar `~/.n8n/credentials.db` dentro do volume (ou `docker volume rm`).
5. `up -d`, completar owner setup de novo, re-importar credenciais.

### Tunnel token

1. Painel Cloudflare → Zero Trust → Networks → Tunnels → senses-contabo →
   Refresh token.
2. Sobrescrever `/root/.cloudflared/senses-contabo.token` com o novo valor.
3. Atualizar `CLOUDFLARE_TUNNEL_TOKEN` em `.env.n8n`.
4. `docker compose restart cloudflared`.

## Debug: tunnel retorna 503

Sintoma: `curl https://n8n.ifops.com.br/healthz` retorna `503`.

Diagnóstico:

```bash
docker logs cloudflared-n8n --tail 100 | grep -i ingress
```

- Se aparecer `No ingress rules were defined`: o `--config` não foi
  passado ou o volume não foi montado. Verificar:
  - `docker inspect cloudflared-n8n` → seção `Mounts` deve ter
    `/etc/cloudflared/config.yml`.
  - O `command` do compose precisa ter `--config /etc/cloudflared/config.yml`
    **antes** do `run`.
- Se aparecer `unable to connect to origin: dial tcp ... refused`: o
  `n8n` está down ou em outra rede.
  ```bash
  docker inspect n8n -f '{{json .NetworkSettings.Networks}}' | jq
  docker exec cloudflared-n8n nc -zv n8n 5678  # (cloudflared distroless, n8n CLI ok)
  ```
